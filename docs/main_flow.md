> 재건축 선도지구 선정용 **제안 동의서** 접수 현황 실시간 모니터링

>    **1단계: 단지 단위 화면**(전체+한양·삼성·두산 3막대)부터 오픈 →

## 1. 목표 (What we are building)

관악타운(A-1구역) 사례처럼, **주민대표단이 엑셀(구글시트)에 입력만 하면**
누구나 링크로 접속해 **단지별·동별 동의율**을 실시간(또는 1시간 이내 간격)으로 확인할 수 있는
웹 대시보드를 만든다.

### 벤치마크(관악타운)에서 확인한 화면 요소
참고 이미지에서 그대로 가져올 구성 요소:

1. **종합 리포트 화면**
   - 전체 동의율 큰 숫자 (예: `29.51%`)
   - 진행 바 + `과반 50%` / `목표 81%` 마커
   - 접수 기간 진행바 (`6/19 → 7/29`, `D-31`, `경과 %`)
   - 핵심 지표 4종: 소유자 수 / 동의자 수 / 미동의 수 / 목표까지 남은 수
   - 단지별(관악타운·부영·성원) 요약 막대 + 동의자/소유자/개동 수
   - 집계 방식 내역: `전자 964 · 서면 64 · 동의 간주 1`
   - 갱신 시각 (`매일 20:00 갱신` 또는 `자동 동기화`)
2. **단지별 상세 화면 (동별 카드)**
   - 동 카드: 동 이름 / 동의율% / 진행바 / `동의자 / 소유자명`
   - 최고 동의율 동에 **👑 왕관** 표시
   - 상가동 별도 그룹
   - 동의율 구간 색상 범례 (8단계, 12.5% 간격)
   - 막대 눈금 = 5% 단위

이 요소들을 우리 3개 단지(한양·삼성·두산) 구조에 맞춰 재구성한다.

## 2. 전체 아키텍처 (How it flows)

```
┌──────────────────────┐
│  주민대표단 (입력 담당)  │   ← 엑셀만 다룰 줄 알면 됨. 프로그램 설치 X
└──────────┬───────────┘
           │ 동의서 접수 시 해당 동의 "동의수" 숫자만 갱신
           ▼
┌──────────────────────┐
│   Google Sheets (5탭) │   ★ 원본 데이터 (Single Source of Truth)
│   - 한양/삼성/두산     │      · 호별 한 줄 + 동의 체크박스 (개인정보 無)
│   - 현황파악(수식집계) │      · 체크 즉시 실시간 집계
│   - 단지정보           │      · 구역명/마감/목표/층수·라인수
└──────────┬───────────┘
           │ (구글 내부에서만 접근)
           ▼
┌──────────────────────┐
│  Google Apps Script   │   ★ JSON API
│  Web App: doGet()     │      · 단지 시트 읽어 동별 숫자로 JSON 반환
│  + CacheService(5~10분)│      · 개인정보 필드 자체가 없음
└──────────┬───────────┘
           │ HTTPS GET (JSON) — 숫자만
           ▼
┌──────────────────────┐
│  대시보드 (정적 웹)     │   ★ 화면. Vercel / Netlify / GitHub Pages
│  - 페이지 로드 시 fetch │      · 5~10분마다 재요청(폴링) → "실시간"
│  - 막대/카드/색상 렌더  │      · 카톡 공유용 리포트 카드 이미지
└──────────┬───────────┘
           │ 링크 공유
           ▼
     주민 누구나 열람 (읽기 전용, 개인정보 없음)
```

### 왜 이 구조인가
- **입력 = 구글시트**: 주민대표단이 새 도구를 배울 필요 없음. 여러 명이 동시에 입력/수정 가능. 자동 백업·수정 이력. 동의서 오면 해당 **호 체크박스만 클릭**.
- **API = Apps Script**: 단지 시트를 읽어 동별 숫자 JSON으로 제공. **개인정보 필드가 없어** 웹 공개가 안전. (→ 6장)
- **화면 = 정적 웹**: 서버 운영·DB 불필요. 무료 호스팅. 트래픽 몰려도 안정적.
- **동기화 주기**: Apps Script 캐시 5~10분 + 대시보드 폴링으로 "즉시~수분 이내" 반영. 요구사항(1시간 이내)을 넉넉히 충족.

## 3. 데이터 모델 (구글시트 설계 — 5탭 · 호별 체크)

> **구현 완료본**: `docs/sheets/Code.gs`(setup·호생성·doGet) + `docs/sheets/구글시트_구축가이드.md`.
> **정확도의 핵심: 호(세대) 1개 = 행 1개.** 그 호가 동의서를 내면 `동의` 체크박스를 켠다.
> **호 목록이 곧 세대수(분모).** 이름·연락처 같은 개인정보는 어떤 탭에도 넣지 않는다(동·호·숫자만).

### 시트 5개 구성
| 탭 | 역할 |
|----|------|
| **현황파악** | 한양·삼성·두산을 **COUNTIF/COUNTIFS 수식으로 실시간 집계**(전체/단지/동별). 체크 즉시 반영 |
| **한양 / 삼성 / 두산** | 동·호가 한 줄씩. `동의`=체크박스, 상단에 🔍 호 검색 패널 + 요약 |
| **단지정보** | 기본정보 + 단지별 공부상 세대수 + 동별 층수·라인수(호 자동생성용) |

### 3-1. 단지 시트(한양/삼성/두산) 열 구조
| A | B | C | D | E |
|---|---|---|---|---|
| 동 | 호 | **동의(체크박스)** | 동의일자 | 비고 |
- 동·호는 `호생성` 함수가 **단지정보의 층수·라인수**로 자동 생성(`호 = 층×100+라인`, 예 1501).
- 우측 `H:I` = 🔍 호 검색(동·호 입력→상태) + 총세대수/동의수/동의율 요약.

### 3-2. `단지정보` 시트
① 기본설정(키/값): 구역명 `평촌 한가람 A-5구역` · 주체 `주민대표단` · 접수마감 `2026-07-29`
· 최소목표 `0.5` · 최종목표 `0.81` · 상가포함 `TRUE` · 단지순서 `한양,삼성,두산` · 마지막갱신(자동)
② 단지별 공부상 세대수(분모 fallback): 한양 952 / 삼성 708 / 두산 436
③ 동별 구성(호생성용): 단지·동·**층수·라인수** — 사용자가 채우면 `호생성`이 호를 깔아준다.

### 3-3. `현황파악` 시트 (전부 수식, 체크 시 즉시 반영)
> ★ **분모(세대수)는 고정값.** 호를 몇 줄 넣든 세대수는 안 변하고, **동의수·동의율만** 변한다.
> (운영 방식이 "동의한 호만 한 줄씩 기록"이므로, COUNTA(호)를 분모로 쓰면 안 됨)
- **단지 세대수** = 공부상 고정값(`VLOOKUP` 아파트세대수+상가세대수). 한양 952·삼성 708·두산 436.
- **단지 동의수** = `COUNTIF(단지!C:C, TRUE)` + 상가 체크 수.
- **동별 세대수** = 단지정보 구성표의 `층수 × 라인수`(고정). 미입력 동은 공란 → 동의율도 공란.
- **동별 동의수** = `COUNTIFS(INDIRECT(단지&"!A:A"), 동, …!C:C, TRUE)`.
- **전체** = 세 단지 합. **동의율** = 동의수/세대수.

### 3-4. 집계·표시 규칙
- **상가**: 각 단지 1개(`상가` 시트, A열=단지) → 하단 별도 카드. `상가포함=TRUE`면 전체 합산 포함.
- **동별 동의율**: 층수·라인수를 넣은 동만 %표시(분모 고정), 미입력 동은 동의수만 표시.
- 색상 8단계: `[0,12.5,25,37.5,50,62.5,75,87.5,100]%` · 막대 눈금 5%.
- **목표까지 남은 수** = `ceil(총세대수 × 0.81) − 동의수합`.

> **동기화(함수)**: 체크박스 → `현황파악` 수식이 **즉시** 반영(트리거 불필요).
> 웹 대시보드는 `doGet` API(5~10분 캐시)가 동일 숫자(분모 고정)를 제공.

## 4. 동기화 방식 (3가지 옵션 비교)

> 개인정보가 없으므로 어느 방식이든 유출 위험은 없다. 선택 기준은 **캐싱·설정 관리·안정성**.

| 방식 | 실시간성 | 캐싱/설정 | 난이도 | 비용 | 추천 |
|------|---------|----------|-------|------|------|
| **A. Apps Script Web App(JSON API)** | 5~10분(캐시) | ◎ 캐시·`설정`시트 병합 | 중 | 무료 | **★ 추천** |
| B. 시트를 CSV로 게시 → 프론트가 직접 파싱 | 즉시 | △ 캐시 없음, 파싱 처리 | 하 | 무료 | 대안 가능 |
| C. 별도 백엔드 서버 + DB + 크론 | 원하는 대로 | ◎ | 상 | 유료가능 | 과함(불필요) |

### 추천: 방식 A (Apps Script Web App)
- 단지 시트(한양/삼성/두산)+`단지정보`를 합쳐 **한 번의 JSON**으로 제공 → 프론트가 단순해짐.
- `CacheService`로 5~10분 캐싱 → 시트 API 호출 폭주 방지 + "1시간 이내" 요구 충족.
- 배포 한 번이면 URL 고정. 프론트는 이 URL만 fetch.

#### Apps Script 예시 (`Code.gs`)
`동현황` 시트는 이미 동 단위라 집계랄 것도 없다. 그대로 숫자만 읽어 JSON으로 넘긴다.
```javascript
function doGet() {
  const cache = CacheService.getScriptCache();
  const cached = cache.get('summary');
  if (cached) return json(JSON.parse(cached));

  const ss = SpreadsheetApp.getActiveSpreadsheet();
  const rows = ss.getSheetByName('동현황').getDataRange().getValues();
  const header = rows.shift();
  const idx = name => header.indexOf(name);
  const num = v => Number(v) || 0;

  // 동 1개 = 행 1개. 세대수/동의수 등 "숫자만" 추출 (개인정보 필드 없음)
  const 동목록 = rows
    .filter(r => r[idx('동')] !== '')          // 빈 행 제외
    .map(r => ({
      단지:   r[idx('단지')],
      구분:   r[idx('구분')],                   // 아파트 | 상가
      동:     String(r[idx('동')]),
      세대수: num(r[idx('세대수')]),
      동의수: num(r[idx('동의수')]),
      전자:   num(r[idx('전자')]),
      서면:   num(r[idx('서면')]),
      간주:   num(r[idx('간주')]),
    }));

  const payload = {
    갱신시각: new Date().toISOString(),
    설정: readSettings(ss),   // 접수기간/최소목표/최종목표/상가포함 등
    동목록,                   // 숫자만
  };

  cache.put('summary', JSON.stringify(payload), 300); // 5분 캐시
  return json(payload);
}

function json(obj) {
  return ContentService
    .createTextOutput(JSON.stringify(obj))
    .setMimeType(ContentService.MimeType.JSON);
}
```
> 배포: `배포 > 새 배포 > 웹 앱`, 실행 계정 = 나, 액세스 = **전체 공개(익명)**.
> **애초에 개인정보 필드가 없으므로** 익명 공개해도 아무 위험이 없다.

## 6. 개인정보 보호 — 설계상 위험 없음

이 시스템은 **개인정보를 아예 수집·저장하지 않는다.** (소유자 명부 없음 결정)
시트에 존재하는 값은 `단지 / 구분 / 동 / 세대수 / 동의수` 같은 **집계 숫자뿐**이며,
이름·호수·연락처 컬럼 자체가 없다. 따라서:

- JSON API로 나가는 데이터도 **숫자뿐** → 유출될 개인정보가 원천적으로 없음.
- 대시보드를 **완전 공개**해도 안전.

### 그래도 지킬 최소 원칙
1. **동의서 원본(종이/전자)은 별도 오프라인으로 관리.** 이 시스템은 "숫자 카운트"만 다룬다.
   실물 동의서에 담긴 개인정보를 이 시트에 옮겨 적지 않는다.
2. 시트 **편집 권한은 주민대표단 계정만**. (숫자 조작 방지)
3. 대시보드는 **읽기 전용**, 입력 기능 없음.
4. (선택) 세대수가 극히 적은 상가 등은 동의수 공개 시 추정 위험이 있는지 한 번 검토.
   문제되면 상위 그룹에 합산 표기.

> 요약: **다루는 것은 오직 숫자.** 개인정보가 시스템에 들어오지 않으므로 유출 자체가 불가능하다.

## 7. 배포 & 운영

### 배포 단계
1. 구글시트 생성 → `소유자명부`·`설정` 시트 구성, 소유자명부 초기 데이터 입력.
2. Apps Script 작성 → 웹 앱 배포 → URL 확보(테스트: 브라우저로 URL 열어 JSON 확인).
3. 대시보드 코드 작성 → 환경변수에 API URL 넣고 Vercel 배포.
4. 짧은 도메인/링크(예: `bit.ly` 또는 Vercel 서브도메인) 만들어 카톡 공지.

### 운영
- **입력 담당(주민대표단)**: 동의서 접수 시 `동현황` 시트에서 해당 동의 `동의수`를 갱신(누적 카운트). (선택) 전자/서면/간주 내역도 기입.
  - 1단계에선 동별 세대수가 없어도 되지만, **동의수는 동별로 입력**해두면 나중에 동별 세대수만 채워도 동별 화면이 바로 완성됨.
- **갱신 주기**: 자동(5~10분 캐시). 별도 수동 조작 불필요.
- **모니터링**: 대시보드 상단 "갱신시각"으로 정상 동작 확인.
- **마감(7/29) 대비**: 막판 입력 폭주 대비 입력 담당 2명 이상 분담, 캐시 주기 1~2분으로 단축 가능.
